# Not-So-Localhost

Local services accessible anywhere via Cloudflare Tunnel, with Keycloak authentication and Traefik routing.

## Prerequisites

- Docker + Docker Compose
- Copy `.env.example` to `.env` in each service directory and fill in secrets
- AWS credentials for S3 backup (`backup/.env`)
- (macOS only) [ttyd](https://github.com/tsl0922/ttyd) for web terminal — `brew install ttyd && brew services start ttyd`

## Cloudflare Setup

Before starting the stack, set up Cloudflare Tunnel and DNS:

1. **Authenticate** — `cloudflared tunnel login` (saves credentials to `~/.cloudflared/`).
2. **Create a tunnel** — `cloudflared tunnel create <name>` — creates a tunnel and downloads its credentials JSON (e.g. `<uuid>.json`). Copy that JSON to `cloudflared/<uuid>.json`.
3. **Configure the tunnel UUID** — Open `cloudflared/config.yml` (generated by config-gen) and verify the tunnel UUID matches. The template `cloudflared/config.yml.tmpl` has a placeholder — update it with your tunnel UUID before starting.
4. **Route DNS** — `cloudflared tunnel route dns <uuid> *.YOUR_DOMAIN` — this creates a wildcard CNAME `*.YOUR_DOMAIN` pointing to your tunnel.

The stack uses these subdomains — all covered by the wildcard DNS route:

| Subdomain          | Service                                     |
|--------------------|---------------------------------------------|
| `auth.YOUR_DOMAIN` | Keycloak admin console (no auth)            |
| `t.YOUR_DOMAIN`    | Web terminal (via oauth2-proxy)             |
| `apps.YOUR_DOMAIN` | App Registry dashboard (via oauth2-proxy)   |
| `*.YOUR_DOMAIN`    | Catchall — deployed apps (via oauth2-proxy) |

### Corporate Proxy / SSL Inspection

If you're behind a corporate firewall that does SSL inspection, the cloudflared container needs your corporate CA certificate to verify Cloudflare's edge TLS connections:

```bash
# macOS
security find-certificate -a -p -c "NICE" > cloudflared/ca-bundle.pem

# Windows (PowerShell) — export your corporate CA from Cert Manager to PEM
```

The file is gitignored and injected at runtime by the cloudflared container's entrypoint. Without it, the tunnel will fail with `x509: certificate signed by unknown authority`.

## Quick Start

```bash
# 1. Set your domain
echo 'DOMAIN=your-actual-domain.com' > .env

# 2. Start the stack (config-gen generates configs automatically)
docker compose up -d
```

On macOS, ttyd must be running separately (`brew services start ttyd`) for web terminal access at `t.YOUR_DOMAIN`. See ttyd section below for alternatives.

## Architecture

```text
Cloudflare Tunnel → Traefik:80
  ├── auth.YOUR_DOMAIN  → keycloak:8080
  ├── t.YOUR_DOMAIN     → oauth2-proxy → host.docker.internal:7681 (ttyd)
  ├── apps.YOUR_DOMAIN  → oauth2-proxy → registry:7272
  └── *.YOUR_DOMAIN     → oauth2-proxy → registry:7272 (catchall)
```

- Cloudflare Tunnel terminates TLS and forwards to Traefik on port 80.
- Traefik uses the file provider with dynamic config in `traefik/dynamic/`.
- oauth2-proxy provides forward-auth middleware; protected routes authenticate via Keycloak.
- Wildcard DNS `*.YOUR_DOMAIN` is routed through Cloudflare Tunnel (`cloudflared tunnel route dns`).
- ttyd runs natively on macOS (not in Docker); Traefik proxies to `host.docker.internal:7681`.
- A Dropbear SSH server (`terminal` service) runs in Docker on port 2222 for SSH-based terminal access.
- Config templates (`infra.yml.tmpl`, `config.yml.tmpl`) are rendered at startup by a one-shot `config-gen` container using `envsubst`.

## Hostnames

| Hostname           | Service                                    | Auth     |
|--------------------|--------------------------------------------|----------|
| `auth.YOUR_DOMAIN` | Keycloak admin console                     | N/A      |
| `t.YOUR_DOMAIN`    | ttyd (Mac shell via oauth2-proxy)          | Required |
| `apps.YOUR_DOMAIN` | App Registry (via oauth2-proxy)            | Required |
| `*.YOUR_DOMAIN`    | Catchall — App Registry (via oauth2-proxy) | Required |

## Adding a New App

Register the app at the Registry dashboard (`apps.YOUR_DOMAIN`). The Registry writes the Traefik route to `traefik/dynamic/managed.yml` automatically. No code changes, tunnel edits, or Docker Compose modifications needed.

## Web Terminal (ttyd)

### macOS
ttyd runs natively on the host — Docker Desktop for Mac can't run containers in the host network namespace (no `--network host`). Launch it with:
```bash
brew install ttyd && brew services start ttyd
```
Access at `t.YOUR_DOMAIN` via oauth2-proxy.

### Windows / other platforms
Options (pick one):
- **WSL2** — Install ttyd inside WSL and access via `host.docker.internal:7681`.
- **Windows OpenSSH Server** — Enable the built-in SSH server and use an SSH terminal client instead of ttyd. The Dropbear SSH container (`terminal` service) already provides SSH on port 2222 if you prefer containerized access.
- **Containerized terminal** — Run `tsl0922/ttyd` as a Docker service (note: this gives you a shell inside the container, not the host).

## Backup

The `backup` service dumps registered databases to S3 on a schedule. Configure AWS credentials and bucket in `backup/.env`.

## Configurable Domain

Set `DOMAIN` in `.env` at the repo root. All runtime config is generated from this single variable:

- `docker-compose.yml` uses `${DOMAIN}` — Docker Compose substitutes from `.env` at startup.
- `traefik/dynamic/infra.yml` and `cloudflared/config.yml` are generated from `.tmpl` templates by the `config-gen` container at startup using `envsubst`.
- `traefik/dynamic/managed.yml` is auto-generated by the registry at runtime — never edit it directly.
- The registry reads `DOMAIN` from its own `.env` and exposes it via `/api/config`; the frontend loads it dynamically.

Upstream changes merge cleanly: tracked files contain `${DOMAIN}` placeholders, never a real domain.
